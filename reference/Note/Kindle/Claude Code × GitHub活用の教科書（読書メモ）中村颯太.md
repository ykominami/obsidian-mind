# 2章 Claude Codeを「コードを書くAI」と誤解してはいけない
## ３つの役割
1.全体の把握
2.糸の言語化
3.判断材料の提示
## 参謀として使う
- 作業は人間が責任を持つ
- 整理と提案はClaude Codeが担当
- 判断基準はGitHub上に記録
## GitHubで情報を集約
- ファイル構成
- Issueや議論の履歴
- Pull Requestの差分
## 実例
### リポジトリ全体から業務内容を説明させる
- 新規参加メンバーに向け、Claude Codeにリポジトリの目的、想定ユーザー、変更が入りやすい箇所を説明させる
- 得られた内容をREADMEに反映させれば、オンボーディング資料として再利用できる
### 過去のIssue・PRから設計思想を抽出
- Claude Codeで横断的に読み込ませることで、なざその設計が選ばれたのか、どの議論を経たかを整理した文章として抽出できる
### 今すぐできること
Claude Codeに「目的・設計意図・注意点」を説明させ、その内容をREADMEに追加する
# 3章：業務に耐えるGitHubリポジトリの設計思想
- 整理されている感覚が得られない-.「誰のために、何を前提としてリポジトリを作っているか」があいまいなまま運用されていることにある
- 単なるコード置き場->業務の背景や判断基準を共有する場所
- 「察してもらう設計」->「説明しておく設計」
## READMEは「説明書」ではなく、「業務ルール集」
- 「何をするか」->「どう判断するか」
	- このリポジトリが扱う業務の範囲
	- 判断に迷った時の優先順位
	- 変更を加える際に必ず確認すべき前提
	- 過去に採用しなかった方針と、その理由
## 情報を分けておくという設計思想
- docs: 業務として正式に残す情報
- prompts: Claude Codeへの依頼内容や指示テンプレート
- logs: 試行錯誤や失敗の記録
- 「確定情報」と「揺らぎのある情報」の分離
## 実例1:個人事業主の業務ナレッジ整理
- 未来の自分に説明する装置
- 日々の判断理由、方針->Claude Codeが過去の思考を踏まえた相談相手になる
- README: 業務方針
- docs: 案件ごとの考え方をまとめる
- logs: 失敗やまとめを書き残す
## 実例2:小規模チームでの業務共有
- 口頭で補っていた背景情報を文章として残す->Claude Code経由でも状況を把握できる
- Issue, PR 「何を変えたか」だけでなく、「なぜそう判断したか」を意識して残すことが重要
## 最初に決めておくべき3つの基準
1. 業務資産として残す情報の範囲
2. 残さなくてもよい情報の線引き
3. 判断に迷った時の優先基準
- これらをREADMEに明文化
## 完璧な構成を目指さない
# 4章　ドキュメント業務の組み換え
- 「生成」ではなく「整流」に価値を置く
- 作成した後に「どう使われ続けるか」
