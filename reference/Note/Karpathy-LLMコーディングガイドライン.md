# Karpathy — LLMコーディングガイドライン

→ **コーディング前に考える** 前提を述べる。不確かなときは聞く。決して推測しない。

→ **シンプルさを優先** 問題を解決する最小限のコードを書く。誰も求めていない抽象化はしない。

→ **外科手術的な変更** リクエストに関係ないコードには触れない。変更したすべての行が、リクエストされた内容に遡れるようにする。

→ **目標指向の実行** 曖昧な指示を検証可能な成功基準に変える。1行も書く前に。

## 拡張ルール（CLAUDE.md 適用版）

> 元の4原則を実運用向けに拡張したルール群。

- **ルール5 — モデルは判断が必要なときだけ使う**: 分類・下書き・要約・抽出はLLM。ルーティング・リトライ・ステータスコード処理はコードで。
- **ルール6 — トークン予算は参考値じゃない**: タスクごと4,000トークン、セッションごと30,000トークン。予算が近づいたら要約して再構築。
- **ルール7 — ぶつかりを見つけて平均化しない**: 2パターンが矛盾するなら1つを選ぶ（より新しい・よりテストされた方）。矛盾するパターンを混ぜない。
- **ルール8 — 書く前に読む**: コード追加前に、exports・immediate callers・共有ユーティリティを読む。
- **ルール9 — テストは「意図」を見る**: テストはWHY（なぜその挙動が大事か）をコードに埋め込む。ビジネスロジックが変わっても失敗しないテストは間違い。
- **ルール10 — 重要ステップごとにチェックポイント**: やったこと・検証できたこと・残ってることをまとめる。説明できない状態から先に進まない。
- **ルール11 — 反対でもコードベースの慣習に合わせる**: Conformance > taste。慣習が害だと思ったら明示的に出す。黙って別ルートに分岐しない。
- **ルール12 — 大きく失敗させろ**: 何かがスキップされたなら「Completed」は間違い。不確実性は隠さず表に出す。
